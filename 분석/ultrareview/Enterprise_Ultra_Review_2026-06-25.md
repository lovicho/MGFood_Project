## 1. Executive Verdict

현재 프로젝트는 **실행 가능한 쇼핑몰 제품**이라기보다, 순대 ODM 온오프라인 쇼핑몰 구축을 위한 **기획/제안/벤치마킹 패키지**에 가깝다. 방향성은 좋지만, 엔터프라이즈 등급으로 실제 구축 계약 또는 개발 착수에 들어가기에는 핵심 통제 항목이 아직 빠져 있다.

| 항목 | 판정 |
|---|---|
| 전체 준비도 | 42 / 100 |
| 현재 단계 | Pre-MVP / Proposal Due Diligence |
| 기업용 구축 착수 | No-Go, P0 게이트 완료 후 착수 권장 |
| 제안/협상 자료 활용 | Go, 단 계약/소유권/PG/개인정보 조항 보강 필요 |
| 가장 큰 리스크 | 실행 가능한 소스/인프라 부재, 결제/개인정보/계약 통제 미완성 |

핵심 결론은 단순하다. **제안서의 비전은 엔터프라이즈급을 지향하지만, 저장소의 실물 산출물은 아직 엔터프라이즈급 구현물의 증거를 제공하지 않는다.** 따라서 다음 단계는 디자인을 더 화려하게 만드는 것이 아니라, 범위, 소유권, 결제 검증, 개인정보, 운영 인수 조건을 문서와 계약으로 잠그는 것이다.

## 4. Critical Findings

### CR-01. 실행 가능한 제품 소스가 없음

제안서는 `Next.js 14+`, `React`, `Strapi`, `PostgreSQL`, PG 연동을 목표 스택으로 제시한다. 근거는 `외주업체제안/순대_ODM_온오프라인_쇼핑몰_구축_제안서.md:60`, `:63`, `:65`, `:68`이다. 그러나 저장소에는 해당 스택의 실제 소스, 설정, 패키지 매니페스트, DB 스키마, API 명세, CI/CD가 없다.

영향:

- 개발 견적, 일정, 품질 보증, 보안 심사가 문서 추정에 머문다.
- 외주사 또는 내부 개발자가 동일한 결과물을 재현할 기준이 없다.
- "엔터프라이즈급"이라는 표현을 검증할 수 있는 빌드 산출물이 없다.

P0 조치:

- `README`, `PRD`, `SOW`, `system architecture`, `data model`, `API contract`, `acceptance criteria`를 먼저 고정한다.
- 실제 구축이면 최소 Next.js/TypeScript 앱 스캐폴드, lockfile, `.env.example`, 테스트 러너, CI 워크플로를 저장소에 추가한다.
- 빌더형 플랫폼을 선택할 경우에도 관리자 권한, 데이터 내보내기, 도메인 소유권, PG 정산 권한을 계약 산출물로 만든다.

### CR-02. 결제 아키텍처가 엔터프라이즈 통제 수준에 미달

문서에는 `PG 결제 데이터 변조를 막기 위해 웹훅(Webhook) 검증 방식을 도입`한다고 되어 있다(`외주업체제안/순대_ODM_온오프라인_쇼핑몰_구축_제안서.md:72`). 그러나 주문 상태 모델, 서버 간 승인 확인, 웹훅 서명 검증, 멱등성 키, 금액 대조, 재고 차감 타이밍, 환불/부분취소, 장애 복구 규칙은 없다.

영향:

- 프론트 콜백 신뢰, 중복 승인, 재고 불일치, 주문 미완료, 결제 완료 후 화면 정지 같은 장애가 발생할 수 있다.
- PG 추가 비용과 구현 범위가 계약서에서 불명확해질 수 있다. 기존 분석도 PG 추가 비용을 위험으로 기록했다(`분석/경쟁업체_제안서_및_시장_분석_보고서.md:18`, `:103`).

P0 조치:

- 주문 상태를 `CREATED`, `PAYMENT_PENDING`, `PAYMENT_CONFIRMED`, `PAID`, `FULFILLING`, `SHIPPED`, `CANCELLED`, `REFUNDED`, `FAILED`처럼 명시한다.
- 결제 성공 처리는 PG 서버 승인 API와 DB 주문 금액을 서버에서 대조한 뒤에만 완료한다.
- 웹훅은 서명 검증, 재시도, 멱등성 키, 중복 이벤트 처리, 감사 로그를 필수로 둔다.
- 환불, 부분취소, 정산, 현금영수증/세금계산서, 실패 알림 흐름을 별도 명세로 만든다.

### CR-03. 계약 소유권과 락인 위험이 실제 통제 산출물로 전환되지 않음

기존 분석은 2년 약정, 위약금, 도메인/호스팅/소스 소유권, 락인 위험을 잘 지적했다(`분석/경쟁업체_제안서_및_시장_분석_보고서.md:61`, `:65`, `:66`, `:100`-`:103`). 하지만 현재 저장소에는 이를 실제 계약 체크리스트, 검수표, 인수 조건으로 변환한 파일이 없다.

영향:

- "무료 제작" 또는 "저가 패키지" 계약에서 도메인, 호스팅, 소스, 관리자 권한이 외주사 계정에 묶일 수 있다.
- 2년 이후 이전 비용이나 유지보수 독점 비용이 실제 총소유비용을 키울 수 있다.

P0 조치:

- 계약서 부속서로 `소유권/인수/해지/위약금/데이터 반출/계정 양도` 체크리스트를 만든다.
- 도메인 등록자, DNS, 호스팅, PG, 네이버/카카오/구글/인스타그램/유튜브 계정은 더푸른식품 명의로 개설한다.
- 외주사 산출물 인수 기준에 원본 디자인, 소스, DB 백업, 이미지 원본, 계정 목록, 비밀번호 인계 절차를 포함한다.

### CR-04. 개인정보 및 전자상거래 준수 설계가 빠져 있음

제안서에는 `사업자 등록증을 첨부`하는 B2B 무료 샘플 신청 프로세스가 있다(`외주업체제안/순대_ODM_온오프라인_쇼핑몰_구축_제안서.md:113`). 이는 개인정보와 사업자 정보, 경우에 따라 대표자명/연락처/주소/파일 업로드를 처리한다. 그러나 개인정보 처리방침, 수집 동의, 보관 기간, 파기, 접근권한, 파일 업로드 보안, 위탁 처리, 제3자 제공, 전자상거래 표시 의무가 구체화되어 있지 않다.

영향:

- 개인정보보호법, 전자상거래법, 통신판매업 신고, 고객 응대/환불 정책 측면의 리스크가 있다.
- 사업자등록증 파일 업로드는 악성 파일, 과다 보관, 내부 권한 오남용 위험을 만든다.

P0 조치:

- 개인정보 처리항목, 목적, 보관기간, 파기, 위탁사, 권리 행사 절차를 문서화한다.
- 파일 업로드는 확장자/크기 제한, 백신 스캔, 비공개 스토리지, 서명 URL, 다운로드 감사 로그를 적용한다.
- 쇼핑몰에는 사업자 정보, 통신판매업 신고, 교환/환불/배송, 미성년자 구매, 고객센터, 약관, 개인정보 처리방침을 필수 화면으로 둔다.

### CR-05. AI 이미지와 HACCP 표현이 허위/과장 표시 리스크를 만들 수 있음

AI 이미지 프롬프트는 HACCP 인증 공장, HACCP certification mark, 더푸른 순대 패키지 mockup을 생성하도록 되어 있다(`외주업체제안/나노바나나_AI_이미지_프롬프트_명세서.md:40`, `:45`, `:53`). 실제 인증서, 제조 파트너, 허가된 마크 사용 범위가 확정되지 않은 상태에서 이런 이미지를 홍보물로 쓰면 소비자 오인 가능성이 있다.

영향:

- 식품 표시광고, 인증 마크 사용, 제조원/판매원 표기, 원산지/영양성분/알레르기 표시 측면에서 법적 검토가 필요하다.
- AI 생성 공장 이미지가 실제 제조시설 사진처럼 보이면 신뢰도 하락과 분쟁 위험이 커진다.

P0 조치:

- AI 이미지는 `연출 이미지`, `컨셉 이미지`, `실제 생산시설 아님` 같은 표기 기준을 만든다.
- HACCP 마크와 인증 문구는 실제 인증서, 인증 범위, 제조원 허가, 표시 가능 문구를 확인한 뒤 사용한다.
- 제품 패키지 이미지는 식품 표시사항 검토 후 확정한다.

## 5. Major Findings

### MJ-01. 태스크는 완료 표시와 실제 증거가 불일치함

`.zto/dashboard.md`는 완료 및 검증을 선언하지만, `plan/tasks.md`는 모든 작업이 미완료 체크박스로 남아 있다. 또한 `verify-convergence.js` 실행을 요구한다(`plan/tasks.md:17`)고 되어 있으나 해당 스크립트는 저장소에서 발견되지 않았다.

조치:

- 완료된 태스크와 미완료 태스크를 실제 파일 증거 기준으로 재정렬한다.
- 검증 스크립트가 필요하면 실제로 추가하고, 아니라면 문구를 제거한다.
- `Definition of Done`을 파일 단위로 둔다.

### MJ-02. 제품 요구사항이 기능 목록 수준에 머물러 있음

현재 문서는 결제, PWA, SEO/AEO/GEO, ODM 홍보를 제시하지만 사용자 여정, 관리자 여정, B2B/B2C 권한, 주문/재고/배송/반품, 쿠폰/정기구독, 도매 견적, 콘텐츠 승인 흐름이 세분화되어 있지 않다.

조치:

- MVP 범위를 `브랜드 소개`, `제품 카탈로그`, `문의/샘플`, `B2C 주문`, `관리자`, `CMS`, `SEO`로 나눠 사용자 스토리와 인수 기준을 작성한다.
- B2B 도매 기능은 MVP에서 견적/샘플 신청까지만 둘지, 사업자 전용 가격/주문까지 둘지 결정한다.

### MJ-03. 보안 아키텍처와 운영 통제가 부족함

Next.js/Strapi/PostgreSQL을 쓸 경우 필요한 인증, RBAC, 관리자 MFA, 비밀관리, 감사 로그, 백업, WAF, rate limit, CSP, 에러 추적, 접근 로그, 취약점 패치 정책이 없다.

조치:

- 최소 보안 기준을 OWASP ASVS L1/L2 수준으로 정의한다.
- 관리자 계정은 MFA, 최소권한, IP 제한 또는 조건부 접근을 적용한다.
- 비밀값은 `.env` 파일 공유가 아니라 Vercel/AWS/GitHub Secrets 같은 관리형 저장소에 둔다.

### MJ-04. PWA와 오프라인 QR 결제가 과장되기 쉬움

제안서는 오프라인 매장 QR 결제와 PWA를 결합한다고 한다(`외주업체제안/순대_ODM_온오프라인_쇼핑몰_구축_제안서.md:48`, `:114`). 하지만 Service Worker 캐싱 규칙, 결제 요청 네트워크 바이패스, 재연결 동기화, 실패 상태 UI, 영수증 재발급이 정의되지 않았다.

조치:

- PWA는 1차 MVP에서 "홈 화면 추가, 정적 자산 캐싱" 수준으로 제한한다.
- 결제 API, 주문 API, 개인정보 제출 API는 Service Worker 캐시에서 제외한다.
- 오프라인 QR 결제는 "오프라인에서도 결제 가능"이 아니라 "QR을 통해 모바일 웹 결제 페이지로 이동"으로 정확히 표현한다.

### MJ-05. SEO/AEO/GEO 전략이 실행 계획으로 내려오지 않음

문서에는 JSON-LD, FAQ, 언론/특허/HACCP 인증기관 링크가 언급되어 있다(`외주업체제안/순대_ODM_온오프라인_쇼핑몰_구축_제안서.md:87`, `:105`). 그러나 실제 키워드 전략, URL 구조, sitemap, robots, canonical, schema 유형, 콘텐츠 캘린더, 출처 검증 프로세스가 없다.

조치:

- 핵심 키워드를 `순대`, `병천순대`, `순대국 밀키트`, `HACCP 순대`, `도매 순대`, `ODM 식품 제조`처럼 intent별로 나눈다.
- 제품/레시피/제조공정/인증/FAQ/납품문의 페이지의 schema.org 타입을 미리 정의한다.
- GEO는 과장된 용어보다 신뢰 가능한 출처, 인증서, 언론 보도, 제조원 정보의 일관성이 핵심이다.

### MJ-06. HTML 제안서의 접근성과 검색 메타가 부족함

HTML 제안서는 `title`, `charset`, `viewport`, `lang`은 갖추고 있으나 description, Open Graph, JSON-LD가 없다(`외주업체제안/순대_ODM_온오프라인_쇼핑몰_구축_제안서.html:2`-`:6`). Accordion은 `div` 클릭 이벤트로만 동작하며 키보드 접근성, `button`, `aria-expanded`, `aria-controls`가 없다(`:548`-`:567`).

조치:

- 프레젠테이션 HTML이라도 `<meta name="description">`, OG 태그, print CSS를 추가한다.
- Accordion header는 `button`으로 바꾸고 aria 속성과 키보드 포커스 스타일을 넣는다.
- 외부 Google Fonts 의존이 필요하면 장애 시 fallback을 확인하고, 배포용은 self-host 또는 preload 정책을 검토한다.

### MJ-07. 벤치마킹 미러 데이터의 거버넌스가 없음

참조 홈페이지 미러에는 login, admin, nonce가 포함된 URL 형태의 파일명, 수많은 이미지/폰트/HTML, 인코딩이 깨진 로그가 포함되어 있다. 이는 분석 목적에는 쓸 수 있지만 운영 저장소에 계속 넣기에는 부담이 크다.

조치:

- `참조홈페이지`는 분석 원본 보관소로 분리하고, 제품 개발 저장소에서는 제외한다.
- 필요한 내용은 `extracted_site_data.json`처럼 요약/정규화한 데이터만 남긴다.
- 저작권/개인정보/크롤링 정책에 맞춰 보관 기간과 삭제 정책을 정한다.

### MJ-08. 테스트와 품질 게이트가 없음

현재 실행 가능한 코드가 없기 때문에 테스트도 없다. 하지만 제안 수준에서도 품질 게이트는 만들 수 있다.

필수 게이트:

- 문서 검수: PRD, SOW, IA, 디자인, 법무/개인정보, 결제 명세 승인
- 구현 검수: unit, integration, e2e, visual regression, accessibility, Lighthouse
- 운영 검수: backup restore, monitoring alert, incident runbook, 관리자 권한 점검
- 보안 검수: secret scan, dependency audit, SAST, CSP, rate limit, admin MFA

### 엔터프라이즈 최소 아키텍처

- Frontend: Next.js + TypeScript, server rendering 또는 static generation
- CMS: Sanity/Strapi, 콘텐츠 승인 워크플로
- Commerce: PG 서버 승인, 주문 상태 머신, 환불/정산 모듈
- Data: PostgreSQL 또는 managed commerce DB, 백업/복구 정책
- Storage: 비공개 원본 이미지 스토리지 + CDN 변환
- Security: MFA, RBAC, WAF/rate limit, CSP, secret management
- Observability: error tracking, access log, order/payment audit log, uptime monitor
- Compliance: 약관, 개인정보 처리방침, 통신판매업, 표시광고 검수

## 10. Final Recommendation

이 프로젝트는 지금 당장 외주 계약서에 서명하거나 개발 착수금을 집행하기보다, **P0 문서 6개를 먼저 작성하고 검수 회의를 통과한 뒤 진행**하는 것이 맞다. 특히 더푸른식품이 장기적으로 온라인 판매와 브랜드 자산을 직접 가져가려면, 첫 계약에서 도메인/계정/소스/데이터 소유권을 놓치지 않는 것이 기술 스택 선택보다 더 중요하다.

가장 안전한 다음 액션은 다음 두 가지 중 하나다.

1. 빠른 상용화: 더푸른식품 명의의 빌더형 쇼핑몰 계정으로 MVP를 만들고, 외주사는 디자인/촬영/콘텐츠만 수행하게 한다.
2. 맞춤 개발: Next.js 기반 저장소를 새로 만들고, 결제/개인정보/운영 요구사항을 P0 문서로 잠근 뒤 작은 MVP부터 구현한다.

현재 산출물은 협상과 방향 설정에는 충분히 가치가 있다. 다만 엔터프라이즈급 실행으로 가려면 이제부터는 "멋진 제안"이 아니라 "검증 가능한 통제"를 쌓아야 한다.
